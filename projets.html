Cloud Design Patterns : cluster MySQL haute dispo sur AWS

Cloud & DevOps

Implémentation d’un cluster MySQL complet sur AWS EC2 en appliquant des design patterns cloud (Proxy et Gatekeeper). L’infrastructure inclut un manager MySQL, deux workers, un proxy HTTP pour le routage des requêtes et une chaîne de sécurité Gatekeeper + Trusted Host, le tout déployé et benchmarké automatiquement.

  • Architecture : VPC dédié, groupes de sécurité par rôle, manager/replicas MySQL, instance Proxy, Gatekeeper exposé en entrée, Trusted Host avec politiques IPtables par défaut en DROP.
  • Patterns implémentés : Proxy pattern avec trois modes de routage (DIRECT_HIT, RANDOM, CUSTOMIZED suivant le ping et la charge des nœuds) et Gatekeeper pattern pour contrôler le flux entrant, filtrer les requêtes et isoler le Trusted Host.
  • Sécurité : VPC isolé, security groups fins, IPtables sur le Trusted Host (DROP par défaut, SSH restreint, port applicatif limité au VPC), chaîne Gatekeeper→Trusted Host→Proxy→Cluster.
  • Benchmark : campagnes Sysbench automatisées, comparaison des trois modes (latence moyenne, CPU CloudWatch, succès à 100%). Direct Hit maximise la vitesse brute, Random rééquilibre la charge, Customized optimise la stabilité et l’utilisation CPU (< 20% sur tous les nœuds).
  • Stack : AWS EC2/VPC/Security Groups/CloudWatch, Python (boto3, requests), MySQL, Sysbench, IPtables, Flask pour les endpoints HTTP.

Estimation bayésienne de la durée de fonctionnement des véhicules de chantier

Actuariat · Bayésien

Projet de recherche M2 (2025) : développement d’un modèle hiérarchique bayésien pour estimer la durée de fonctionnement de véhicules de chantier dans un contexte assurantiel. Le modèle combine une loi de Weibull pour les durées de fonctionnement et une loi exponentielle pour les durées de maintenance, en intégrant des covariables chantier et maintenance (sol, météo, distance, techniciens…).

  • Objectif : prédire la durée de vie résiduelle, quantifier l’incertitude et proposer une méthodologie de tarification pour les contrats d’assurance (facteur de risque, durée de couverture, prime ajustée).
  • Approche : modèles linéaires & réseaux de neurones pour les fonctions de lien g₁ et g₂, estimation fréquentiste (MLE) puis inférence bayésienne (Metropolis–Hastings adaptatif), validation sur données simulées.
  • Insights clés : importance dominante de la nature du sol, rôle plus modéré de la maintenance précédente, possibilité de catégoriser les chantiers par niveau de risque et d’ajuster les primes en conséquence.
  • Stack : Python (pandas, NumPy, PyMC3 / MCMC maison), stats bayésiennes, analyse de survie (Weibull, exponentielle), visualisation.

MapReduce sur AWS : WordCount & recommandations sociales

Cloud · Data Engineering

TP avancé de calcul distribué sur AWS autour de deux axes : (1) comparer les performances de différents frameworks (Hadoop, Spark, Linux simple) sur un WordCount classique et (2) implémenter un système de recommandation d’amis dans un réseau social à l’aide de MapReduce (détection de “friends-of-friends” et mutual friends).

  • Expériences WordCount : mêmes datasets passés sur Hadoop (HDFS), Spark et un pipeline Linux cat | tr | sort | uniq pour mesurer l’impact de l’overhead cluster vs single node. Résultat : Linux ~13× plus rapide que Hadoop sur de petits fichiers, Hadoop ~3× plus rapide que Spark dans ce setup précis.
  • Recommandations sociales : algorithme MapReduce qui génère des recommandations d’amis à partir des amis d’amis : phase Map pour émettre les paires de users partageant un ami commun, phase Reduce pour compter les mutual friends et produire un top des suggestions par utilisateur.
  • Points intéressants : filtrage des amis directs pour ne pas les recommander, gestion des secondes relations, tri par nombre d’amis communs, résultats lisibles pour des IDs d’utilisateurs donnés.
  • Stack : Hadoop / HDFS, Spark, MapReduce, AWS, Python, scripts de benchmark, manipulation de graphes sociaux.
Voir le repo “MapReduce”
